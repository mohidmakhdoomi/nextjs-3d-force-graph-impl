## Executive Summary

Implement the specification's selected atomic Next 15 / React 19 backport in
two phases:

1. reverify the six exact artifacts and their compatibility metadata, capture
   the pre-change audit/tree baseline, update `package.json` and
   `package-lock.json` together under Node `22.23.1` / npm `10.9.8`, add a
   narrow dependency contract, and inspect the resulting peer and lock trees;
2. qualify the committed graph through clean installation, static checks,
   production build/start, the existing real-WebGL Chromium smoke, the full
   manual interaction matrix, and final audit-path review.

The manifest and lockfile remain one rollback unit. No application component,
Next 16, 3D, Tailwind, direct PostCSS, lint-stack, or React Compiler change is
planned. Raw audit JSON remains transient local/CI evidence; durable totals,
path-by-path applicability conclusions, and the nested PostCSS disposition
will be written only to the review document during the Review phase.

#### Implementation Details

1. Record the pre-upgrade commit as the rollback baseline. Before any dependency

   - `.nvmrc` is `22.23.1`;
   - `node --version` is `v22.23.1`;

   - the worktree has no unexplained tracked dependency-file changes.
2. Before manifest modification, capture:
   - `npm ls --all --json`;
   - `npm audit --json` with its original exit code;
   - `npm audit --omit=dev --json` with its original exit code; and
   - validator output from `scripts/validate-audit-report.mjs` for both audit
     reports.
   Advisory exits are evidence, while malformed JSON or registry/tool failures
   block the phase.
3. Query the registry for every exact target using `npm view` and record
   publication, version, dist-tags, engines, peer dependencies, and material
   direct dependencies. Explicitly inspect:
   - `next@15.5.20` support/backport status, React/DOM peer range,
     `@next/env`, `styled-jsx`, SWC optional dependencies, and nested PostCSS;
   - `@next/eslint-plugin-next@15.5.20`;
   - `react@19.2.7`;
   - `react-dom@19.2.7`, its React peer, and scheduler dependency;
   - `@types/react@19.2.17` and its `csstype` dependency; and
   - `@types/react-dom@19.2.3` and its React-types peer.
   Confirm Node `22.23.1` satisfies every relevant engine and that React
   `19.2.7` is stable. If a selected artifact is unpublished, no longer the
   intended tag/support line, incompatible, or superseded in a way covered by
   the specification's drift rule, stop for architect confirmation rather than
   substitute autonomously.
4. Update only these existing direct entries, preserving dependency
   classification and exact syntax:

   ```text
   dependencies:
     next       15.5.20
     react      19.2.7
     react-dom  19.2.7

   devDependencies:
     @next/eslint-plugin-next  15.5.20
     @types/react              19.2.17
     @types/react-dom          19.2.3
   ```

   Regenerate `package-lock.json` once with npm `10.9.8`. Do not use
   `--force`, `--legacy-peer-deps`, `npm audit fix`, or an `overrides` entry.
5. Extend `tests/toolchain.test.mjs` with behavior-level dependency contracts:
   - the six manifest values equal the verified exact group and contain no
     range operators;
   - React equals React DOM and Next equals its lint plugin;
   - matching root lockfile dependency/devDependency values are already
     covered by the existing root-synchronization assertion;
   - target lock records resolve to the intended versions; and
   - exactly one `node_modules/react` runtime record exists in the lockfile.
   Keep provenance and full transitive attribution as review procedures rather
   than encoding the complete npm graph in brittle tests.
6. Remove the installed tree through the normal clean-install workflow and run
   plain `npm ci`. Inspect `npm ls next @next/eslint-plugin-next react react-dom
   @types/react @types/react-dom postcss --all`; any `invalid`, `UNMET`,
   peer warning, duplicate React runtime, prerelease, or mismatch blocks the
   phase.
7. Review the complete package-lock diff:
   - root manifest metadata and lockfile version;
   - every changed, added, and removed package record;
   - version, resolved URL, integrity, engines, peers, and optional metadata;
   - Next, `@next/env`, `styled-jsx`, every native SWC optional package, and
     `next > postcss`;
   - React, React DOM, scheduler, React types, and `csstype`;
   - all registry sources and introduced majors; and
   - the parent range responsible for every material transitive move.
   Reject unexplained churn or any new git/file/HTTP/unknown registry source.
8. Capture and validate post-change full and production audits with original
   exits. Compare each before/after advisory and every root-to-affected-package
   path, including paths that npm coalesces into one advisory record. Preserve
   the actual `next > postcss@8.4.31` installed path and
   GHSA-qx2v-qp2m-jg93/CVE-2026-41305 for the final review. Do not commit the
   generated JSON or claim that the audit is clean.

#### Rollback Strategy

If qualification fails because of the package baseline, revert the entire
Phase 1 dependency commit, including manifest, lockfile, and contract
assertions, then clean-install and rerun the old baseline. Do not selectively
pin a transitive, change the 3D/CSS/lint stack, or retain a new manifest with an
old lockfile. A qualification-only tracking commit may be reverted separately
because it contains no dependency state.

#### Risks

- **Risk**: Static/build checks pass while an imperative graph interaction
  regresses.
  - **Mitigation**: Require both the real-WebGL smoke and the full manual matrix.
- **Risk**: Software WebGL makes browser actions slow or flaky.
  - **Mitigation**: Follow the existing baseline lesson: pause continuous
    rotation before later actions, prove pointer receipt when forcing the first
    pause, and keep later interactions ordinary and observable.
- **Risk**: A compatibility failure tempts unrelated cleanup.
  - **Mitigation**: Stop, classify the failure, and limit any returned fix to
    the approved Next/React boundary; defer 3D, CSS, and lint modernization.
- **Risk**: Final audit output changes due to registry/advisory updates.
  - **Mitigation**: Preserve timestamps and original exits, validate structure,
    reconcile by advisory/path, and distinguish measurement drift from lockfile
    drift.
